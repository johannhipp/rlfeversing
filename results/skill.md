# Reverse Engineering Skill

## Techniques

- For public/open-source targets, classify open vs closed from primary sources first. If shell networking is restricted, use the GitHub connector or web access to fetch the repo README, license, and specific source files instead of trying to clone.
- For open-source monorepos, identify the actual shipped harness package before analyzing architecture. Root docs may describe the product broadly while the real execution loop, tool registry, and delegation code live in a subpackage such as `extensions/cli/`.
- If a target has both an old archived repo and a newer active repo, verify which one matches the shipped artifact before analyzing architecture. Public repo lineage can drift across org renames or full rewrites.
- For open-source agent harnesses, distinguish prompt-defined tool contracts from provider-native function calling. Public docs that show XML-like or tag-based tool syntax are strong evidence that the harness owns tool parsing/execution in prompt space, even if "native tool call" exists as an optional provider feature.
- When reverse-engineering agent harnesses, look for the execution loop first, then the tool registry, then any delegation or subagent module. That usually answers most of the useful questions faster than reading docs.
- If a target instructs you to wrap interactive commands with `timeout` but the host does not have GNU `timeout`, use a safe substitute such as `python3 -c 'import subprocess; subprocess.run(..., timeout=10)'`.
- For npm-installed proprietary CLIs, inspect the installed package before deeper RE: `which <tool>`, follow the symlink, then read `package.json`, `LICENSE*`, `README*`, and any shipped `.d.ts` files.
- Treat shipped declaration files as primary evidence. They often reveal the harness contract more clearly than a minified runtime bundle: tool names, subagent inputs/outputs, async/background behavior, permission modes, and extension points.
- For closed-source targets, classify "public repo" versus "public shell" early. A public GitHub repo can still be only issues/docs while the real harness ships as a binary tarball.
- For Bun-compiled CLIs, `strings` often proves the packaging model (`bun build --compile`) even when the original TS/JS sources are not shipped.
- Probe non-interactive subcommands before `--help`. Some TUIs hang on top-level help, but specific subcommands such as `permissions list --builtin` or `<subcommand> --help` return immediately and reveal architecture.
- Treat timeout wrappers as part of the experiment, not just a safety guard. For Amp, `amp --help` succeeded under a Perl `alarm` wrapper after other timeout methods looked hung, while several subcommands still failed under the same guard.
- If a CLI emits a startup network error before printing help text, do not discard the probe. Capture stderr and stdout together; the later usage text can still prove command taxonomy, server/headless modes, and programmatic output formats.
- If the tool stores local session history, inspect transcript JSONL early. Closed-source harnesses may persist raw `tool_use` / `tool_result` records, which reveal the actual tool taxonomy and execution loop more directly than docs.
- For open-source Python harnesses, inspect package `__init__.py` exports before treating a module as part of the live product. This is a fast way to distinguish supported coder/agent modes from deprecated or orphaned experiments.
- For open-source harnesses that contain old tool-calling experiments, verify liveness through the active registry or factory path before crediting them as real harness capabilities. A deprecated tool-schema module that is not exported or selected by the main factory should be treated as dead or secondary.
- For open-source harnesses with many role or mode names, trace the factory/clone path before calling them subagents. If the "handoff" reuses the same IO/history/file context inside one process, that is usually mode specialization rather than a real orchestrator/worker system.
- For edit-centric open-source harnesses, distinguish parsed shell snippets from true tool calls. If the model emits shell blocks inside plain text and the harness later asks the user before execution, classify that as guarded prompt-parsed side effects, not a first-class autonomous tool API.

## Effective Techniques

- When using the GitHub connector for open-source RE, prefer file fetches by `repository_full_name` + `path` + `ref`. Some connector setups reject `raw.githubusercontent.com` even when the repository itself is reachable.
- For downloadable IDE agents, check the obvious local artifact paths first: PATH CLI, app bundle, and vendor state directories. A missing `which <tool>`, missing `/Applications/<Tool>.app`, and missing documented config/state roots can quickly tell you whether binary-level RE is even possible on the current machine.
- For closed IDE agents, treat that absence check as a deliberate triage step. If the PATH launcher, obvious app bundle, and vendor state roots are all missing, binary RE is blocked for environmental reasons and you should pivot quickly to docs/hooks/config-schema RE instead of broad filesystem spelunking.
- For closed IDE agents, combine the docs/install surface with the vendor GitHub org surface. If the official flow is binary-first and the public org only exposes integrations, plugins, or demos rather than the core editor, that is strong evidence the harness itself is closed-source.
- If the target CLI is proprietary, inspect the installer before chasing the public repo. Install scripts often reveal exact binary hosts, version pins, architecture splits, and bundled helper binaries.
- If the target has a public GitHub repo, verify both license and repo scope before calling it open-source. A repo that is all-rights-reserved and mainly contains docs/plugins/examples is not the same thing as an open-source harness.
- If GitHub code search is failing on a public repo, use docs or package metadata to recover exact file paths and fetch those files directly. For Cline, the docs and `package.json` were enough to jump straight to the tool registry and subagent specs.
- For Bun/TS-adjacent agent CLIs, `strings` on the shipped binary is useful for confirming the runtime family and surfacing embedded command/tool names even when source is unavailable.
- For Bun/TS agent CLIs, inspect adjacent app-state directories like `~/.<tool>/` early. They may contain bundled `node_modules`, cached prompts, and `.d.ts` files that reveal the real harness contract faster than binary reversing.
- If the target persists local thread/session JSON, mine that corpus early. Transcript files can reveal real tool names, model IDs, subagent/delegation primitives, cross-thread memory tools, and persisted tool artifacts even when the runtime is compiled and closed-source.
- Treat shipped SDK/plugin typings as primary evidence for agent harness behavior. They often expose subagent message parts, permission taxonomies, tool hooks, session APIs, and PTY capabilities directly.
- For proprietary Node-distributed CLIs, inspect the bundle banner itself. Comments like `All rights reserved` or `Want to see the unminified source?` are strong evidence that the shipped file is a minified proprietary build rather than a source release.
- For open-source agent harnesses, inspect the helper that executes streamed tool calls, not just the top-level chat loop. Files like `handleToolCalls` or `streamChatResponse.helpers` often reveal whether tool execution is sequential or parallel, how permissions are enforced, and whether the harness has real batching behavior.
- For open-source harnesses with terse source files, fetch the raw docs pages alongside the tool-spec source. The docs often expose the exact allowed/forbidden subagent capabilities, approval behavior, and user-trigger conditions more clearly than the TypeScript prompt spec alone.
- If GitHub raw fetches are awkward or lose structure, switch to the blob page and use line-oriented search/open tooling there. For Cline, that was the fastest way to recover exact source-backed claims like `contextRequirements` gates and per-parameter prompt slots.
- Agent CLIs may not return quickly for `--help` or `-v`. Always use a hard timeout wrapper. If `timeout` is unavailable, use a local equivalent such as Python `subprocess.run(..., timeout=10)`.
- On macOS, `perl -e 'alarm shift; exec @ARGV' 10 <cmd> ...` is a simple timeout substitute when GNU `timeout` is missing.
- Before debugging a local interception proxy, verify whether the environment allows loopback listeners at all. A one-line `socket.bind(("127.0.0.1", 0))` probe can quickly distinguish a sandbox networking policy failure from a bug in the proxy implementation.
- Check the tool’s local state directory early. For Factory, `~/.factory/` exposed sessions, tool artifacts, plugins, and subagent definitions that were more informative than the public repo.
- Look for user-authored subagent definitions on disk (`droids`, `agents`, similar folders). These often reveal the exact delegation primitive, tool taxonomy, and safety model used by the harness.
- If the bundle is minified but still JavaScript, search it for high-signal terms instead of reading top-to-bottom: `subagent`, `task_progress`, `handoff`, `allowedAgentTypes`, `mcp`, `worktree`, and tool names like `WebFetch` or `Grep`.
- On closed Bun bundles, grep `strings` for protocol and event identifiers, not just tool names. Session update labels, role names, and permission enums can prove whether the harness has orchestrator/worker roles, streamed tool telemetry, or structured approvals.
- If a subagent schema includes explicit async launch, output-file tracking, permission mode, or worktree isolation, classify it as real harness-level delegation rather than prompt theater.
- Product pages can leak real harness claims even when source is unavailable. Treat mentions of tool families, non-interactive output formats, approval boundaries, and model-per-subagent routing as useful architectural evidence.
- For closed-source IDE harnesses, hook docs are often better than feature pages. Pre/post hook event names expose the concrete action surface and blocking points for reads, writes, terminal commands, MCP calls, prompts, and worktree setup.
- If hook docs mention transcript-export hooks or on-disk audit artifacts, treat those paths and sample schemas as primary RE evidence. They can reveal internal step types, planner/tool sequencing, and persisted trajectory formats without needing the binary.
- If the sandbox blocks writes under the real home directory, rerun the target with `HOME=$PWD/.tmp-<tool>-home` to get cleaner startup traces and writable local state. The tradeoff is that you may lose existing auth/session state.
- For proxy work, a failed connection to `127.0.0.1:<port>` can still be useful evidence. If debug logs mention the custom base URL and then emit connection retries, that is enough to prove the redirect hook is live even when you cannot start a local listener.
- For open-source harnesses that already expose an OpenAI-compatible `apiBase` or equivalent provider endpoint override, proxy the model provider path instead of searching for a proprietary service URL. Continue fits this pattern: generate a temporary config that points `apiBase` at the local proxy and capture provider traffic there.
- If the target binary is missing and the sandbox blocks loopback binds, still finish the proxy task by shipping a self-contained proxy repo, an outside-sandbox runner, and offline tests. Record both blockers explicitly so later agents know the missing live capture is environmental rather than a gap in the redirect design.
- When live capture is blocked, still verify the proxy repo itself offline: run the unit tests, parse the proxy CLI with `--help`, lint the runner with `sh -n` and `bash -n`, then probe `command -v <target-cli>` and a raw `127.0.0.1` bind separately. That cleanly distinguishes code issues from missing binaries and sandbox networking limits.
- For provider-backed harnesses, do not assume a custom base URL means HTTP-only traffic. Codex `0.117.0` attempted `ws://127.0.0.1:18456/v1/responses` first and then retried `http://127.0.0.1:18456/v1/responses`, so a realistic capture proxy needs both WebSocket and HTTP handling.
- For proprietary CLIs with both login and API-key auth paths, a repo-local fresh `HOME` is useful for separating "redirect works" from "auth state missing". The auth error itself can clarify whether requests would have been sent if credentials were available.
- For proxy capture wrappers around proprietary CLIs, keep the existing authenticated `HOME` by default and make clean-state `HOME` overrides opt-in. A fresh home is good for redirect sanity checks, but it often suppresses the real authenticated traffic you actually need to capture.
- If a redirected proprietary client disables auxiliary features on non-first-party hosts, debug logs may reveal the exact override you need for faithful capture. Claude Code, for example, disables optimistic Tool Search on custom `ANTHROPIC_BASE_URL` hosts unless `ENABLE_TOOL_SEARCH=true`.
- For npm-installed CLIs, inspect the global package before assuming the executable is proprietary. `which <tool>`, `ls -l $(which <tool>)`, `file $(which <tool>)`, then read the installed `package.json` and `README.md`.
- An open-source npm CLI may still install as a launcher plus packaged native binary. Treat the local package as packaging evidence, then follow `repository.url` and any `repository.directory` metadata to fetch the real source tree from the public repo.
- Read the launcher script before reversing the platform binary. For Codex, `bin/codex.js` immediately revealed that the public npm package is just a Node shim that selects a packaged native binary per target triple.
- Even when the project is open-source, run `strings` on the shipped native binary. Codex leaked internal Rust source paths, collaboration event names, MCP modules, and dynamic tool call identifiers that confirmed real harness capabilities faster than a full source dive.
- Feature-flag listings are high-signal for agent harnesses. Codex exposed `multi_agent`, `apps`, `plugins`, `shell_tool`, `unified_exec`, and MCP-related tool plumbing via `codex features list`.
- Do not over-trust feature-flag listings when they conflict with live probes. Codex still attempted WebSocket traffic on `/v1/responses` even while `codex features list` marked `responses_websockets` and `responses_websockets_v2` as removed.
- After classifying a target as open-source, pivot quickly to source-guided RE. For Codex, `codex-rs/core/src/tools/spec.rs` and the concrete handler files were the fastest way to confirm which tool calls and subagent flows are real.
- If live interception is blocked by the sandbox, still finish the proxy deliverable by adding an outside-sandbox runner script that sources `.env`, launches the proxy, runs one safe non-interactive harness command through the redirected base URL, and writes artifacts into the target directory.
- For open-source provider-driven CLIs, prefer intercepting the provider base URL over hunting for a product-specific backend. If the harness exposes flags like `--openai-api-base`, that is usually the cleanest proxy seam.
- If the proxy seam is known but the local binary is missing, still build the proxy and the capture wrapper. Record both blockers separately: "redirect hook exists" versus "artifact absent on this machine".
- Do not blindly reuse a capture-stub proxy for provider-backed open-source harnesses. If the harness needs a real model response to complete a non-interactive run, adapt the stub into a forwarding reverse proxy instead of returning generic 200 JSON.
- For provider-traffic capture, consider forcing `Accept-Encoding: identity` upstream by default. Readable JSON/SSE logs are usually more valuable for RE than preserving compressed response bodies byte-for-byte.
- In capture wrappers, poll a local `/health` endpoint before launching the harness. That cleanly separates "proxy never bound" from later auth/provider/runtime failures.
- For Python-distributed CLIs, do not assume the console-script name is the only runnable form. If docs or packaging support `python3 -m <module>`, make the capture wrapper accept that path too instead of hard-failing on a missing standalone binary.
- More generally, do not assume the harness is launched as a single bare executable. Capture wrappers should support multi-word command prefixes from environment variables so `python3 -m ...`, `uvx ...`, `npx ...`, `bun ...`, or other package-manager launchers can be swapped in without editing the script.
- When proxy startup is suspected but live capture is still blocked, run a minimal raw socket bind probe on the target host/port. A standalone `bind(('127.0.0.1', ...))` failure cleanly distinguishes sandbox listener restrictions from bugs in the proxy code.
- For provider-capture proxies, add offline tests for the exact provider endpoint shapes your harness may hit, such as `/v1/responses` and `/v1/chat/completions`. That keeps the redirect seam verifiable even when you cannot capture live traffic in the current sandbox.
- Do not classify a CLI probe as "hung" from an empty first read when you are driving it through an async exec harness. For Codex `0.117.0`, the first 1-second poll for `exec --help` and `mcp --help` returned nothing, but both commands completed successfully within the 10-second wrapper on a later poll.
- For proprietary IDE agents, product docs often reveal more harness detail than marketing pages: look for planning agents, retrieval subagents, tool-call budgets, approval modes, local config paths, and which operations execute locally on the user's machine.
- For IDE harnesses that document `AGENTS.md`, rules, skills, or workflows, treat those docs as architectural evidence rather than just user guidance. They often expose prompt layering, scoping rules, progressive disclosure, and whether reusable procedures are automatic or manually invoked.
- For webpack-bundled JS CLIs, enumerate internal module paths before reading code blobs. A quick `rg -o '"./src/[^"]+"' ... | sort -u` often exposes the real harness structure faster than generic keyword search.

## Dead Ends / Caveats

- Do not treat a public repo as proof the harness itself is open-source. Verify whether it contains the actual runtime source or only docs, marketplace assets, and install instructions.
- Do not assume one timeout wrapper's behavior generalizes perfectly. Claude Code `--help` behaved differently under Python timeout versus a Perl alarm wrapper on the same machine; keep the hard timeout and record the wrapper used when results differ.
- A shell-side `curl` DNS failure is an environment limitation, not a finding about the target. Record it, then continue with web/docs-based RE instead of over-interpreting the failure.
- Anonymous GitHub code search is low-yield. Once `strings` or manifests reveal probable repo paths, fetch raw file URLs directly instead of depending on GitHub’s unauthenticated search UI.
- Broad filesystem scans across macOS support directories are often noisy or blocked by privacy controls; prefer documented vendor-specific paths such as `~/.codeium/windsurf/*` when product docs expose them.
- Do not assume `--help` is side-effect free. Opencode attempted a `models.dev` network fetch before completing several help/debug-style commands.
